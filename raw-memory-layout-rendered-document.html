Rust layout caveat RenderedDocument has the default Rust representation. Field order, padding, and total size are not an ABI contract. This diagram is a source-order mental model for a 64-bit build.
Why the object is mostly pointers Each String is a three-word handle: pointer, length, capacity. The UTF-8 content lives elsewhere on the heap.
Clone behavior Cloning RenderedDocument allocates fresh heap buffers for the four String fields, but the &'static str still points at the same static bytes.
Source value render_markdown constructs this struct after Comrak produces HTML and local links are rewritten.
Stack words The visible stack object contains handles and small scalars, not the Markdown or HTML bytes themselves.
Heap buffers Owned string buffers are movable allocations. Capacity may exceed length; unused cells are shown as dashed padding.
Static trust model trust_model is a fat pointer into static read-only storage, used by the frontend trust boundary.

Sources: src-tauri/src/markdown.rs defines RenderedDocument and construction paths; src-tauri/src/trusted_preview.rs defines TRUST_MODEL. Offsets assume conventional 64-bit Rust String and &str word layouts for explanation, not a stable Rust ABI.
